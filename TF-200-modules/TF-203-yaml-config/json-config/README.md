**Course**: TF-203 YAML-Driven Configuration

## 📋 Overview

While TF-203 focuses on YAML-driven configuration, JSON is equally common in real-world Terraform workflows — especially when consuming API responses, CI/CD pipeline outputs, or working with teams that prefer JSON. This supplement covers `jsondecode()`, `jsonencode()`, and when to choose JSON vs YAML.

---

## 🎯 Learning Objectives

By the end of this section, you will be able to:

- ✅ Parse JSON files using `jsondecode()`
- ✅ Generate JSON output using `jsonencode()`
- ✅ Drive resource creation from JSON configuration files
- ✅ Choose between JSON and YAML for different use cases
- ✅ Use `jsonencode()` for inline policy documents
- ✅ Handle JSON from external data sources

---

## 🔑 JSON vs YAML: When to Use Each

| Aspect | JSON | YAML |
|--------|------|------|
| **Readability** | Verbose, more punctuation | Clean, minimal syntax |
| **Comments** | ❌ Not supported | ✅ Supported (`#`) |
| **Multiline strings** | Awkward (`\n`) | ✅ Natural (block scalars) |
| **API responses** | ✅ Universal format | Rare |
| **CI/CD artifacts** | ✅ Common | Less common |
| **Human editing** | Harder | Easier |
| **Tooling support** | Universal | Very common |
| **Terraform functions** | `jsondecode()` / `jsonencode()` | `yamldecode()` / `yamlencode()` |

**Rule of thumb**:
- Use **YAML** for human-authored configuration files
- Use **JSON** when consuming machine-generated data (APIs, CI/CD, scripts)

## 📚 `jsondecode()` — Reading JSON Files

### Basic Usage

```hcl
# Read and parse a JSON file
locals {
  config = jsondecode(file("${path.module}/config.json"))
}
```

### Example: JSON Config File

`config.json`:
```json
{
  "environment": "production",
  "app_name": "hashi-training",
  "servers": [
    { "name": "web-01", "role": "frontend", "port": 80 },
    { "name": "api-01", "role": "backend",  "port": 8080 },
    { "name": "db-01",  "role": "database", "port": 5432 }
  ],
  "tags": {
    "project": "hashi-training",
    "owner": "platform-team"
  }
}
```

`main.tf`:
```hcl
locals {
  config  = jsondecode(file("${path.module}/config.json"))
  servers = local.config.servers
  tags    = local.config.tags
}

# Create a config file per server
resource "local_file" "server_configs" {
  for_each = {for s in local.servers : s.name => s}

  content  = <<-EOT
    name = "${each.value.name}"
    role = "${each.value.role}"
    port = ${each.value.port}
    env  = "${local.config.environment}"
  EOT
  filename = "${path.module}/configs/${each.key}.conf"
}
```

## 📝 `jsonencode()` — Generating JSON Output

`jsonencode()` converts a Terraform value to a JSON string. This is extremely useful for generating policy documents, configuration files, and API payloads inline.

### Basic Usage

```hcl
# Generate a JSON string from a Terraform map
output "config_json" {
  value = jsonencode({
    environment = var.environment
    version     = var.app_version
    servers     = var.server_names
  })
}
```

### Example: Inline Policy Document

```hcl
# Generate a JSON policy document inline (no separate file needed)
resource "local_file" "iam_policy" {
  content = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["s3:GetObject", "s3:ListBucket"]
        Resource = [
          "arn:aws:s3:::${var.bucket_name}",
          "arn:aws:s3:::${var.bucket_name}/*"
        ]
      }
    ]
  })
  filename = "${path.module}/iam-policy.json"
}
```

### Example: Generating Structured Output

```hcl
variable "servers" {
  type = list(object({
    name = string
    role = string
    port = number
  }))
  default = [
    { name = "web-01", role = "frontend", port = 80 },
    { name = "api-01", role = "backend",  port = 8080 },
  ]
}

# Generate a JSON manifest of all servers
resource "local_file" "server_manifest" {
  content = jsonencode({
    generated_at = timestamp()
    environment  = var.environment
    server_count = length(var.servers)
    servers      = var.servers
  })
  filename = "${path.module}/server-manifest.json"
}
```

## 🔄 Combining JSON and YAML

You can mix both in the same configuration:

```hcl
locals {
  # Load infrastructure config from YAML (human-authored)
  infra_config = yamldecode(file("${path.module}/infrastructure.yaml"))

  # Load server inventory from JSON (machine-generated by CI/CD)
  server_inventory = jsondecode(file("${path.module}/inventory.json"))

  # Merge both sources
  all_servers = concat(
    local.infra_config.servers,
    local.server_inventory.servers
  )
}
```

## 🔬 Using `terraform console` to Test JSON Functions

```bash
terraform console
```

```hcl
# Encode a map to JSON
> jsonencode({"name" = "web", "port" = 80})
"{\"name\":\"web\",\"port\":80}"

# Decode a JSON string
> jsondecode("{\"name\":\"web\",\"port\":80}")
{
  "name" = "web"
  "port" = 80
}

# Pretty-print with jsonencode (Terraform formats it)
> jsonencode({
    servers = ["web", "api", "db"]
    count   = 3
  })
"{\"count\":3,\"servers\":[\"web\",\"api\",\"db\"]}"

# Access nested values after decoding
> jsondecode("{\"servers\":[{\"name\":\"web\",\"port\":80}]}").servers[0].name
"web"
```

## ⚠️ Common Pitfalls

### Pitfall 1: JSON Doesn't Support Comments

```json
// ❌ This is NOT valid JSON — comments are not allowed
{
  "environment": "prod"  // This will cause a parse error
}
```

```yaml
# ✅ YAML supports comments — use YAML for human-authored configs
environment: prod  # This is fine
```

### Pitfall 2: JSON Number Types

```hcl
locals {
  data = jsondecode("{\"port\": 80, \"enabled\": true}")
}

# port is a number, enabled is a bool — Terraform preserves types
# data.port = 80 (number)
# data.enabled = true (bool)
```

### Pitfall 3: `jsonencode()` Sorts Keys Alphabetically

```hcl
> jsonencode({"z" = 1, "a" = 2, "m" = 3})
"{\"a\":2,\"m\":3,\"z\":1}"  # Keys are sorted!
```

This is usually fine but can matter for some APIs that are sensitive to key order.

## 🧪 Hands-On Lab

### Lab: JSON-Driven Server Configuration

```bash
mkdir tf203-json-config
cd tf203-json-config
```

**Step 1**: Create `servers.json`:

```json
{
  "environment": "dev",
  "servers": [
    {
      "name": "web-01",
      "role": "frontend",
      "port": 80,
      "enabled": true
    },
    {
      "name": "api-01",
      "role": "backend",
      "port": 8080,
      "enabled": true
    },
    {
      "name": "db-01",
      "role": "database",
      "port": 5432,
      "enabled": false
    }
  ],
  "tags": {
    "project": "hashi-training",
    "owner": "platform-team"
  }
}
```

**Step 2**: Create `main.tf`:

```hcl
terraform {
  required_version = ">= 1.14"
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}

locals {
  # Parse the JSON config file
  config = jsondecode(file("${path.module}/servers.json"))

  # Filter to only enabled servers
  enabled_servers = {
    for s in local.config.servers : s.name => s
    if s.enabled
  }
}

# Create config files for enabled servers only
resource "local_file" "server_configs" {
  for_each = local.enabled_servers

  content  = <<-EOT
    # Server: ${each.value.name}
    role = "${each.value.role}"
    port = ${each.value.port}
    env  = "${local.config.environment}"
  EOT
  filename = "${path.module}/configs/${each.key}.conf"
}

# Generate a JSON manifest of the deployment
resource "local_file" "deployment_manifest" {
  content = jsonencode({
    generated_at    = timestamp()
    environment     = local.config.environment
    total_servers   = length(local.config.servers)
    enabled_servers = length(local.enabled_servers)
    server_names    = keys(local.enabled_servers)
    tags            = local.config.tags
  })
  filename = "${path.module}/deployment-manifest.json"
}
```

**Step 3**: Create `outputs.tf`:

```hcl
output "enabled_servers" {
  description = "Map of enabled servers"
  value       = local.enabled_servers
}

output "manifest_path" {
  description = "Path to the deployment manifest"
  value       = local_file.deployment_manifest.filename
}
```

**Step 4**: Run the lab:

```bash
terraform init
terraform apply -auto-approve
cat deployment-manifest.json
ls configs/
terraform destroy -auto-approve
```

## ✅ Checkpoint Quiz

**Question 1**: Which function parses a JSON string into a Terraform value?
- A) `jsonparse()`
- B) `jsondecode()`
- C) `fromjson()`
- D) `json_read()`

<details>
<summary>Answer</summary>
**B) `jsondecode()`** — Example: `jsondecode(file("config.json"))` reads and parses a JSON file.
</details>

---

**Question 2**: Which function converts a Terraform value to a JSON string?
- A) `tojson()`
- B) `json_encode()`
- C) `jsonencode()`
- D) `stringify()`

<details>
<summary>Answer</summary>
**C) `jsonencode()`** — Example: `jsonencode({"key" = "value"})` produces `"{\"key\":\"value\"}"`.
</details>

---

**Question 3**: What is a key limitation of JSON compared to YAML for configuration files?
- A) JSON doesn't support nested objects
- B) JSON doesn't support comments
- C) JSON can't represent lists
- D) JSON is not supported by Terraform

<details>
<summary>Answer</summary>
**B) JSON doesn't support comments** — This makes JSON less suitable for human-authored configuration files where you want to document why certain values are set. YAML supports comments with `#`.
</details>

## 📚 Key Takeaways

| Function | Purpose | Example |
|----------|---------|---------|
| `jsondecode()` | Parse JSON string → Terraform value | `jsondecode(file("config.json"))` |
| `jsonencode()` | Terraform value → JSON string | `jsonencode({key = "value"})` |
| JSON use case | Machine-generated data, APIs | CI/CD artifacts, API responses |
| YAML use case | Human-authored configuration | Infrastructure config files |
